# AGENTS.md

本文件适用于当前仓库根目录及其所有子目录，用于补充本项目的本地协作与开发规范。若与上层全局规范不冲突，则同时生效；若出现冲突，以用户明确要求为准。

## 1. 项目定位

- 当前仓库是一个 Python RAG 项目，主要链路包括：
  - 文档预处理：`rag/cli/preprocess_docs.py`、`rag/preprocess/document_preprocessor.py`
  - 离线建库：`rag/cli/build_index.py`、`rag/indexing/index_builder.py`
  - 在线检索、重排与答案生成：`rag/cli/answer_query.py`、`rag/retrieval/query_service.py`、`rag/retrieval/retriever.py`、`rag/retrieval/ranking.py`、`rag/retrieval/answer_generator.py`
- `storage/` 目录主要存放本地索引、清洗结果和检查点，默认视为运行产物，不应随意纳入代码提交。
- `tests/` 目录用于放置项目级验证用例；若新增测试，优先保持轻量、可直接运行。

## 2. 交流与文档规范

- 所有交流、代码注释、设计说明、验证结论、提交说明一律使用中文。
- 回答问题时优先给出结论、影响范围和验证结果，避免只描述过程。
- 修改代码时优先补充“为什么这样改”所需的信息，而不是堆砌无意义说明。
- 文档不是一次性产物。凡是涉及功能、流程、架构、配置、运行方式的修改，必须同步检查并更新以下文档中的相关内容：
  - `README.md`
  - `docs/技术分析文档.md`
  - `docs/项目流程文档.md`
- 若某次改动未影响上述文档，也应在交付前显式确认“本次变更无需更新文档”的判断成立。

## 3. 开发修改规范

- 优先做最小侵入式修改，避免与当前任务无关的重构。
- 修改现有模块时，优先沿用仓库已有结构与命名方式，不随意引入新的架构分层。
- 新增日志时必须遵循当前项目约定：
  - 业务日志统一通过 `rag/shared/logging_utils.py` 输出。
  - 进度展示统一通过 `rag/shared/checkpoints.py` 中的 `print_progress()` 输出。
  - 日志默认输出到 `stderr`，进度默认输出到 `stdout`，避免互相覆盖。
- 若修改会影响离线建库、在线检索、预处理等核心链路，必须补充关键节点的可观测性或验证说明。

## 4. 验证规范

- 完成代码修改后，至少执行与改动直接相关的最小验证，不要在未验证的情况下宣称“已完成”。
- 当前仓库优先使用项目虚拟环境执行命令：`.venv/Scripts/python.exe`
- Python 代码改动建议至少选择以下一种或多种验证方式：
  - 运行对应的 `unittest`
  - 运行定向脚本验证
  - 执行 `compileall` 做语法级检查
- 如果因为环境缺失导致无法执行验证，必须明确说明阻塞原因与未验证范围。

## 5. 提交规范

- 提交信息必须遵循“类型前缀 + 中文说明”的格式：

```text
<type>: <中文描述>
```

- `type` 必须根据提交意图选择，常用类型如下：
  - `feat`：新功能、新能力
  - `fix`：缺陷修复、问题修正
  - `docs`：文档、规范、说明更新
  - `refactor`：重构，但不改变外部行为
  - `test`：测试补充或测试修正
  - `chore`：杂项维护、脚本或配置调整
  - `perf`：性能优化
  - `style`：纯格式调整，不涉及逻辑变化
  - `build`：构建流程或依赖管理变更
  - `ci`：持续集成相关变更
  - `revert`：回滚某次提交

- 提交说明必须满足以下要求：
  - 冒号后内容必须使用中文。
  - 描述必须直接说明本次改动目的，禁止使用空泛表述，如“update”“修改一下”“处理问题”。
  - 一次提交只处理一个主题，避免把无关文件混在同一个提交中。

- 合法示例：

```text
fix: 修复改写查询重复召回问题
feat: 增加离线建库关键节点控制台日志
docs: 补充仓库协作与提交规范
test: 增加日志与进度输出兼容性测试
```

- 非法示例：

```text
fix: update logging
feat: add rag log
docs: update
```

## 6. 提交前检查

- 提交前确认 `git status` 中仅包含本次任务相关文件。
- 若发现无关改动，不要擅自回退，先避开或与用户确认。
- 若用户要求提交，则优先按本文件中的提交规范生成提交信息。
