# RAG 优化优先级与排期建议

> 文档目的：沉淀当前项目优化判断，作为后续功能改进、技术治理和版本排期的基线文档。
>
> 适用范围：`rag/` 当前离线建库、在线检索、重排、答案生成与文档预处理链路。
>
> 评估时间：2026-06-04
>
> 评估方式：基于现有代码与项目文档的静态审查，未包含真实回归集跑分结果。

---

## 1. 结论摘要

当前项目已经具备可运行的 RAG 原型能力，链路覆盖：

- 文档预处理
- 离线建库
- 混合召回
- 双阶段重排
- 最终答案生成

但如果要进入“稳定优化、可持续迭代、可排期治理”的阶段，短期最需要优先处理的不是技术栈替换，而是以下四类问题：

1. 文档变动后的旧数据残留与索引一致性
2. 面向隐式业务依赖的召回覆盖不足
3. 所有问题都走完整 RAG 带来的成本、时延与误召回
4. TopK 预算过紧导致的信息过早截断

其中，第 1 类问题是当前最紧急的问题，因为它会直接污染检索结果，使后续所有优化判断失真。

---

## 2. 当前实现基线

基于当前代码，项目已经实现的核心能力如下：

- 预处理支持 `doc`、`docx` 输入，并可抽取 Confluence MHTML 风格图片资源，核心实现见 `rag/preprocess/document_preprocessor.py`
- 离线建库采用 `Markdown -> 语义切分 -> LLM 标注 -> 双路向量化 -> Chroma/BM25/metadata 入库`，核心实现见 `rag/indexing/index_builder.py`
- 在线查询采用 `向量召回 + BM25 + Query Rewrite + 邻居扩展 + 双阶段重排 + 答案生成`，核心实现见 `rag/retrieval/retriever.py`、`rag/retrieval/ranking.py`、`rag/retrieval/answer_generator.py`

这说明当前系统并不是“纯向量召回”，而是已经具备基础混合检索能力。后续优化应以“补短板”为主，而不是推倒重来。

---

## 3. 优先级总览

| 编号 | 问题 | 当前判断 | 优先级 |
| --- | --- | --- | --- |
| 1 | RAG 基于语义相似度召回，容易遗漏加解密、签名等隐式业务依赖 | 已有混合召回，但缺少依赖扩展层 | P0 |
| 2 | 如何处理包含图片、PDF 等附件的文档 | 当前仅覆盖 `doc/docx` 和图片抽取，未覆盖 OCR / PDF 正文治理 | P1 |
| 3 | 向量存储与关键词存储的技术选型 | 当前选型可支撑原型阶段，短期不是主矛盾 | P2 |
| 4 | 文档发生变动时如何处理 | 当前最紧急，存在旧 Markdown 与旧向量残留风险 | P0 |
| 5 | 每个问题都需要经过 RAG 流程吗 | 当前实现默认所有问题都走完整 RAG | P1 |
| 6 | TopK 截断几乎必然导致有用信息丢失，如何缓解 | 当前召回与答案上下文预算偏紧 | P1 |
| 7 | 多轮会话之中如何处理 RAG 和 LLM 之间关系 | 当前本质上仍是单轮问答 | P1 |

补充潜在问题：

- 索引层存在向量库、BM25 和 `metadata.json` 不一致的风险
- 缺少稳定的回归评测集，优化收益难以量化
- LLM 精排只看候选前 400 字，长 chunk 后半段信息可能被忽略

---

## 4. 问题拆解与建议

### 4.1 隐式业务依赖漏召回

**问题描述**

用户问题经常只说主任务，例如“接支付”“接订阅”“做回调”，但真实文档依赖可能分散在签名、验签、加解密、回调幂等、证书管理、错误码、风控等章节中。仅依赖 query 与 chunk 的表层相似度，会导致这类前置约束信息被漏召回。

**当前状态**

当前系统已经具备以下补救手段：

- Query Rewrite
- 正文向量召回
- 摘要向量召回
- BM25 召回
- 邻居扩展

但这些手段仍然主要解决“表达差异”问题，尚未显式解决“业务依赖扩展”问题。

**现有代码依据**

- 查询改写见 `rag/retrieval/ranking.py`
- 混合召回与邻居扩展见 `rag/retrieval/retriever.py`
- 离线标注字段目前主要是 `summary`、`keywords`、`tags`、`type`、`coherence`，见 `rag/indexing/semantic_annotator.py`

**核心风险**

- 能命中主流程，却漏掉关键约束
- 回答看似正确，但实施时缺步骤
- 技术接入类问答的“可执行性”不足

**建议动作**

1. 离线标注新增依赖型字段，例如：
   - `prerequisites`
   - `constraints`
   - `related_api`
   - `related_config`
   - `risk_points`
2. 在线查询增加“业务依赖扩展改写”，不只做同义词改写，还要显式补充签名、回调、权限、异常处理等前置主题
3. 引入“文档级/章节级聚合召回”，避免仅凭单 chunk 相似度决策
4. 建立“接入类问题”的专项评测集，专门检查是否漏掉前置条件

**优先级判断**

P0。因为这类问题会直接影响“答案是否能落地执行”。

---

### 4.2 图片、PDF 等附件处理

**问题描述**

很多业务文档的重要信息并不在正文文字里，而是在：

- 截图
- 流程图
- 架构图
- 扫描版 PDF
- 表格图片

如果这部分内容不进入索引，RAG 只能回答文字化部分，实际覆盖面会明显不足。

**当前状态**

当前预处理链路支持：

- `doc`、`docx`
- Confluence MHTML 风格图片抽取
- 图片引用重写

但尚未实现：

- PDF 输入链路
- OCR
- 图片理解或图片摘要
- 附件与正文之间的结构化关联召回

**现有代码依据**

- 支持的输入后缀目前为 `doc/docx`，见 `rag/preprocess/document_preprocessor.py`
- 当前只抽取图片资源文件，不对图片内容做语义入库，见同文件的附件抽取逻辑

**核心风险**

- 文档存在，系统却回答“查不到”
- 流程图、参数截图、错误页截图无法进入召回
- 扫描 PDF 基本不可用

**建议动作**

1. 先拆成两类处理：
   - 可提取文本 PDF
   - 扫描 PDF / 图片型附件
2. 为扫描件增加 OCR
3. 为流程图、架构图、截图增加 caption 或摘要入库
4. 建立“正文 chunk 命中后补拉附件摘要”的二跳召回机制

**优先级判断**

默认 P1；如果当前知识库中图片/PDF 占比已经较高，应上调为 P0。

---

### 4.3 向量存储与关键词存储技术选型

**问题描述**

需要判断当前 `Chroma + BM25.pkl + metadata.json` 是否需要尽快替换。

**当前状态**

当前方案对本地原型和中小规模离线建库是可用的，优点是：

- 成本低
- 本地可跑
- 改动门槛低
- 调试路径清晰

短期瓶颈并不主要来自底层存储，而是来自上层索引治理和召回策略。

**现有代码依据**

- 向量库存储见 `rag/indexing/storage_indexer.py`
- 查询侧直接读取 Chroma、BM25 与 `metadata.json`，见 `rag/retrieval/retriever.py`

**不建议现在优先替换的原因**

- 现阶段更严重的问题是旧索引残留和召回覆盖率，而不是单机容量
- 在没有回归评测集的前提下换库，很难证明收益
- 过早迁移会拉高工程成本，稀释当前主问题

**建议动作**

1. 短期保留当前选型
2. 先把索引一致性、增量更新、回归评测补齐
3. 当出现以下条件时再评估迁移：
   - chunk 规模显著增长
   - 需要服务化并发读写
   - 需要更强的过滤、分片或在线更新能力

**优先级判断**

P2。短期不建议作为主线工作推进。

---

### 4.4 文档发生变动时如何处理

**问题描述**

文档新增、修改、删除后，如果索引未同步清理，会导致系统继续召回旧内容。

**当前状态**

这是当前最紧急的问题。

预处理阶段会生成新的 Markdown 和来源清单，但不会主动清理旧的无效 Markdown；离线建库阶段会重新读取目录中的 `.md` 文件，并对向量库执行 `upsert`，但没有显式删除旧 `node_id`。

**现有代码依据**

- Markdown 目录只做 `mkdir` 和写入，没有失效文件清理逻辑，见 `rag/preprocess/document_preprocessor.py`
- 文档加载按目录中的 `.md` 全量读取，见 `rag/indexing/document_loader.py`
- Chroma 入库使用 `upsert`，未做旧数据删除，见 `rag/indexing/storage_indexer.py`
- `node_id` 由 `doc_id + chunk_index + text_hash` 生成，文档文本变化后会生成新 ID，见 `rag/shared/checkpoints.py`

**核心风险**

- 已删除源文档仍可能继续参与检索
- 文档更新后，新旧 chunk 同时存在
- 向量库、BM25、`metadata.json` 之间可能不一致
- 所有后续效果评估都会被脏数据污染

**建议动作**

1. 优先提供一个默认安全的“全量重建模式”
2. 在重建前清理以下产物：
   - 失效 Markdown
   - 旧 Chroma collection
   - 旧 BM25 文件
   - 旧 `metadata.json`
3. 中期再补“按 `doc_id` 增量删改”的能力
4. 给重建流程补充显式日志，输出：
   - 新增文档数
   - 删除文档数
   - 更新文档数
   - 最终 chunk 数变化

**优先级判断**

P0。应优先于技术栈升级。

---

### 4.5 每个问题都需要经过 RAG 流程吗

**问题描述**

并不是每一个用户输入都适合直接走检索。

典型不需要或不应立即走 RAG 的情况：

- 问候语
- 元问题，例如“你能做什么”
- 上下文不完整的短追问
- 纯对话型确认

**当前状态**

当前在线查询默认所有问题都走完整链路：

- 检索
- 重排
- 答案生成

**现有代码依据**

- `QueryService.query()` 中固定执行 `retrieve -> generate`，见 `rag/retrieval/query_service.py`

**核心风险**

- 成本和时延上升
- 模糊问题误触发召回
- 多轮追问语义不全时，检索结果容易偏题

**建议动作**

1. 增加轻量问题路由器，至少区分：
   - 非知识库问题
   - 需要检索的问题
   - 需要先做会话补全的问题
2. 对明显不需要检索的问题直接走 LLM
3. 对短追问先结合会话历史重写为完整问题，再决定是否检索

**优先级判断**

P1。建议在 P0 完成后尽快推进。

---

### 4.6 TopK 截断导致信息丢失

**问题描述**

TopK 截断无法完全避免，但可以避免“过早截断”。如果召回阶段预算太小，真正有价值的候选可能在进入重排前就已经被截掉。

**当前状态**

当前默认配置较紧：

- `content_top_k=8`
- `summary_top_k=6`
- `bm25_top_k=8`
- `rewrite_content_top_k=4`
- `rewrite_summary_top_k=3`
- `rewrite_bm25_top_k=4`
- `llm_top_n=10`
- `final_top_n=5`
- `answer_context_top_k=4`

**现有代码依据**

- 召回配置见 `rag/config/retrieval.py`
- 最终候选裁剪见 `rag/retrieval/retriever.py`
- 答案生成上下文裁剪见 `rag/retrieval/answer_generator.py`

**核心风险**

- 召回阶段还没来得及覆盖，答案阶段就已经只剩很少候选
- 多文档分散答案场景下，信息覆盖度不足
- 结构化问题容易只命中局部片段

**建议动作**

1. 把“召回候选预算”和“答案上下文预算”分开治理
2. 先提高召回候选数，再通过重排与压缩控制最终上下文
3. 引入多样性控制，例如：
   - 文档覆盖度约束
   - 章节覆盖度约束
   - MMR 去冗余
4. 对低分差问题使用动态扩召回，而不是固定 TopK

**优先级判断**

P1。它会直接影响整体问答上限。

---

### 4.7 多轮会话中如何处理 RAG 和 LLM 的关系

**问题描述**

多轮对话里，LLM 不应该直接代替 RAG，也不应该让 RAG 去处理全部会话历史。

**当前状态**

当前系统本质上还是单轮模型：

- 改写只看当前 query
- 检索只看当前 query
- 回答只看当前候选

**现有代码依据**

- 改写逻辑见 `rag/retrieval/ranking.py`
- 答案生成只消费本轮 `top_candidates`，见 `rag/retrieval/answer_generator.py`

**建议的职责划分**

- LLM：负责会话理解、指代消解、问题补全、是否需要检索的判断
- RAG：负责提供当前轮次所需证据
- 会话层：负责维护“已确认主题、产品、场景、版本、用户目标”等状态

**建议动作**

1. 增加会话态问题补全模块
2. 区分“继续追问”和“新问题”
3. 不要把完整历史原样塞给检索器，而要先压缩成结构化会话状态

**优先级判断**

P1。适合在问题路由能力之后推进。

## 5. 额外潜在问题

### 5.1 已删除源文档可能继续参与检索

这是第 4 个问题的具体表现，也是最需要优先防止的数据污染问题。

### 5.2 向量库与元数据文件可能不一致

查询侧依赖：

- Chroma
- `bm25.pkl`
- `metadata.json`

如果只更新其中一部分，可能出现召回到了旧 `node_id`，但本地元数据里已经没有对应文本的问题。

### 5.3 缺少回归评测集

当前很多优化只能依赖人工感受，难以支撑稳定排期和版本验收。建议尽快建设一套轻量回归集，至少覆盖：

- 接入类问题
- 配置类问题
- 错误码类问题
- 多轮追问
- 图片/PDF 问题
- 文档更新后的问题

### 5.4 LLM 精排可见信息偏少

当前 LLM 精排只看候选前 400 字。如果关键约束在 chunk 后半段，重排分数可能失真。

## 6. 推荐排期顺序

### 阶段一：数据治理与基线稳定

建议周期：1 到 2 周

目标：

- 解决旧索引残留
- 建立最小回归集
- 保证后续优化有可信基线

建议任务：

1. 增加全量重建模式与清理策略
2. 增加建库阶段的删除/更新统计日志
3. 建立 30 到 50 条核心问题回归集
4. 选取 5 到 10 条高价值问题作为日常冒烟集

### 阶段二：召回覆盖与问题路由

建议周期：2 到 4 周

目标：

- 提高复杂业务问题回答完整度
- 减少不必要的 RAG 调用

建议任务：

1. 增加问题路由器
2. 提高召回预算并增加覆盖度控制
3. 增加依赖型标注字段
4. 增加依赖扩展改写能力

### 阶段三：会话与附件能力

建议周期：3 到 6 周

目标：

- 支持多轮问答
- 覆盖图片和 PDF 资料

建议任务：

1. 增加会话态补全
2. 增加继续追问识别
3. 增加 PDF 文本提取链路
4. 增加 OCR 与图片摘要能力

### 阶段四：存储与服务化升级

建议周期：按业务规模决定

适用前提：

- 数据量显著增长
- 需要在线增量更新
- 需要更强过滤能力或服务并发能力

建议任务：

1. 评估 Qdrant / Elasticsearch / OpenSearch / Milvus
2. 评估索引分层与冷热数据策略
3. 评估服务化后的监控、限流和缓存方案

---

## 7. 建议验收指标

为了让后续优化可以排期和验收，建议引入最小指标体系：

### 7.1 召回侧

- Top10 命中率
- 文档级命中率
- 隐式依赖补全率

### 7.2 回答侧

- 答案完整率
- 引用有效率
- 无法回答判定准确率

### 7.3 工程侧

- 建库耗时
- 查询平均耗时
- 每次查询 LLM 调用次数
- 文档更新后的索引一致性通过率

---

## 8. 最终建议

当前项目下一阶段的主线不建议定义为“换技术栈”，而建议定义为：

**先把数据一致性、召回覆盖、问题路由和回归评测做扎实，再推进附件、多轮会话和底层存储升级。**

推荐执行顺序：

1. 先修文档更新与旧索引残留
2. 再补回归评测集
3. 再提升隐式依赖召回与 TopK 策略
4. 再做问题路由和多轮会话
5. 最后评估附件增强与存储升级
