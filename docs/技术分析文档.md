# 技术分析文档

## 1. 文档目标

本文档从技术实现角度说明当前 RAG 项目的模块划分、职责边界、核心数据流和主要风险。当前版本已反映本次重构后的目录结构、命令入口和完整查询闭环。

## 2. 项目定位

当前项目聚焦于“本地知识库检索增强”场景，核心目标是：

- 将 Word/Confluence 等异构技术文档清洗为统一 Markdown
- 构建面向中文技术文档的多路索引
- 对用户问题提供稳定的召回、重排和最终答案输出
- 为后续服务化和评测闭环保留可扩展结构

相比重构前版本，项目已从“检索基础设施”推进到“可直接回答问题的 RAG 原型”。

## 3. 重构后的总体架构

### 3.1 目录分层

当前代码按职责拆分为五层：

1. `rag/cli/`：命令入口层，只承接参数解析和运行调用
2. `rag/preprocess/`：原始文档预处理层
3. `rag/indexing/`：离线建库层
4. `rag/retrieval/`：在线检索、重排和答案生成层
5. `rag/shared/`：日志、检查点等共享基础设施

这种调整解决了原先“所有 Python 文件平铺在根目录、入口名与职责不一致”的问题。

## 4. 当前已实现能力分析

### 4.1 预处理链路

预处理链路继续保留原有输入治理能力：

- 识别 `doc/docx`
- 尝试直接解析 MHTML 风格内容
- 回退到 `soffice` 转 HTML
- 清理无关标签、隐藏节点和导出噪声
- 抽取图片资源并重写 HTML 引用
- 输出 Markdown 与来源映射清单

技术价值没有变化，但职责归位到单独目录后，后续更容易单独扩展图片处理、增量预处理或新输入格式。

### 4.2 离线建库链路

离线建库仍然采用五段式结构：

1. 加载文档
2. Markdown 语义切分
3. LLM 语义标注
4. 双路向量化
5. 多路入库

本次调整的主要改进点：

- 入口从含义模糊的 `main.py` 改为 [rag/cli/build_index.py](D:/ai/project/rag/rag/cli/build_index.py)
- 预览生成独立为 [rag/indexing/preview_renderer.py](D:/ai/project/rag/rag/indexing/preview_renderer.py)
- `chunks_preview.html` 新增 `node_id` 展示，便于排查召回和引用来源

### 4.3 在线检索、重排与答案生成

当前在线链路已经形成完整闭环：

1. Query Rewrite
2. 三路召回
3. 候选聚合
4. 规则粗排
5. 邻居扩展
6. LLM 精排
7. 最终答案生成
8. 引用来源输出

与重构前相比，最关键的变化是：

- 不再只输出结构化检索结果
- 新增 [rag/retrieval/answer_generator.py](D:/ai/project/rag/rag/retrieval/answer_generator.py)
- 由 [rag/retrieval/query_service.py](D:/ai/project/rag/rag/retrieval/query_service.py) 统一编排检索与回答
- 命令入口改为 [rag/cli/answer_query.py](D:/ai/project/rag/rag/cli/answer_query.py)

### 4.4 模型使用策略

查询侧统一围绕 `qwen3.6-plus` 配置：

- Query Rewrite：`RAG_QUERY_REWRITE_MODEL`
- LLM 精排：`RAG_RERANK_LLM_MODEL`
- 最终答案生成：`RAG_ANSWER_MODEL`

默认值全部为 `qwen3.6-plus`，满足“查询使用还是 qwen3.6-plus”的要求。

## 5. 核心数据结构分析

### 5.1 TextNode

离线切分后的 chunk 节点，关键字段包括：

- `id_`
- `text`
- `metadata`

其中 `metadata` 承载：

- 文档来源
- 文件类型
- `chunk_index`
- `token_size`
- 语义标注字段：`summary`、`keywords`、`tags`、`type`、`coherence`

### 5.2 RetrievalCandidate

这是在线检索阶段的核心聚合单元，统一承接：

- 多路命中明细
- 粗排得分
- 精排得分
- 最终得分
- 是否邻居候选
- 中心 chunk 来源

### 5.3 QueryResponse

这是新增的查询输出结构，统一承接：

- 原始问题
- 改写结果
- 最终答案
- 来源引用
- 全量候选
- Top 候选

这让在线查询从“调试型输出”升级为“业务可消费输出”。

## 6. 技术选型分析

### 6.1 LlamaIndex

继续用于文档与节点抽象，适合作为基础数据结构层。项目核心流程仍由自定义逻辑控制，而不是依赖默认流水线。

### 6.2 Chroma

继续作为本地向量库，适合当前原型与本地验证场景。未来若需要规模化，可平滑替换为 Qdrant、Milvus 等。

### 6.3 OpenAI 兼容接口

标注、改写、精排、答案生成和向量化仍统一走 OpenAI 兼容调用方式，优点是接口统一、切换成本低，但依然要注意：

- Base URL 与模型配置错位
- 兼容服务稳定性
- Key 管理

### 6.4 BM25 与自定义分词

保留路径、下划线、中划线、点号命名和中文复合词，是当前技术文档召回质量的重要保障。

## 7. 可观测性分析

当前可观测性设计已具备较完整基础：

- 阶段级日志
- 进度输出
- 检查点
- 结构化索引产物
- chunk 预览页
- 查询答案引用来源

其中本次新增的 `node_id` 展示价值很直接：

- 预览页可以和 `metadata.json`、查询结果、最终引用一一对应
- 方便人工排查“命中的是哪个 chunk”
- 方便定位邻居扩展与重复召回问题

## 8. 风险与问题

### 8.1 测试覆盖仍偏轻

虽然本次补了预览页 `node_id` 展示测试，但整体仍缺少：

- 检索聚合单测
- 精排解析单测
- 答案生成降级单测
- 端到端回归测试

### 8.2 服务化能力仍不足

当前项目已具备完整问答闭环，但仍是命令行原型，距离生产服务还缺少：

- API 封装
- 请求治理
- 缓存与限流
- 监控告警

### 8.3 LLM 依赖敏感

Query Rewrite、LLM 精排和答案生成依赖兼容 LLM 服务。若配置缺失或接口异常，虽然当前支持部分降级，但答案质量仍会受影响。
