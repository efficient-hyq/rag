# 项目流程文档

## 1. 文档目标

本文档描述当前项目从原始文档输入到最终答案输出的完整执行流程，重点覆盖重构后的命令入口、目录分工、关键产物和排查路径。

## 2. 总体流程图

当前项目可以拆成三条主流程：

1. 文档预处理流程
2. 离线建库流程
3. 在线查询流程

整体关系如下：

```text
原始 doc/docx
    ↓
文档预处理
    ↓
cleaned_markdown
    ↓
离线建库
    ↓
chroma / bm25 / metadata / checkpoints
    ↓
在线查询
    ↓
答案输出 + 来源引用
```

## 3. 文档预处理流程

### 3.1 入口

- [rag/cli/preprocess_docs.py](D:/ai/project/rag/rag/cli/preprocess_docs.py)
- [rag/preprocess/document_preprocessor.py](D:/ai/project/rag/rag/preprocess/document_preprocessor.py)

示例命令：

```bash
.venv/Scripts/python.exe -m rag.cli.preprocess_docs --input-dir "C:/Users/heyuqin/Desktop/RAG_DATA"
```

### 3.2 执行步骤

1. 扫描输入目录下的 `doc`、`docx`
2. 准备 `storage/converted_html`、`storage/cleaned_markdown`、`storage/cleaned_assets`
3. 优先尝试解析 Confluence MHTML 风格文档
4. 无法直接解析时回退到 LibreOffice `soffice` 转 HTML
5. 清洗 HTML 噪声并转换为 Markdown
6. 写入来源映射清单和图片资源清单

### 3.3 输出产物

| 路径 | 说明 |
| --- | --- |
| `storage/converted_html/` | 中间 HTML 产物 |
| `storage/cleaned_markdown/` | 清洗后的 Markdown |
| `storage/cleaned_assets/` | 抽取出的图片资源 |
| `storage/cleaned_markdown/source_manifest.json` | 原始文件与 Markdown 映射清单 |

## 4. 离线建库流程

### 4.1 入口

- [rag/cli/build_index.py](D:/ai/project/rag/rag/cli/build_index.py)
- [rag/indexing/index_builder.py](D:/ai/project/rag/rag/indexing/index_builder.py)

示例命令：

```bash
.venv/Scripts/python.exe -m rag.cli.build_index --docs-dir "./storage/cleaned_markdown" --storage-dir "./storage"
```

### 4.2 执行阶段

#### 阶段 1：加载文档

由 [rag/indexing/document_loader.py](D:/ai/project/rag/rag/indexing/document_loader.py) 负责：

- 从 `cleaned_markdown` 中读取 Markdown 文件
- 使用 `SimpleDirectoryReader` 加载为 `Document`
- 合并 `source_manifest.json` 中的来源元数据

#### 阶段 2：切分 chunk

由 [rag/indexing/markdown_chunker.py](D:/ai/project/rag/rag/indexing/markdown_chunker.py) 负责：

- 按标题、段落、表格、代码块进行语义切分
- 对超大文本块做二次拆分
- 补充 `chunk_index`、`token_size`
- 根据文档路径、chunk 序号、文本哈希生成稳定 `node_id`

同时会写入：

- `storage/checkpoints/chunks.jsonl`
- `storage/chunks_preview.html`

其中 `chunks_preview.html` 会展示：

- `chunk_index`
- `node_id`
- 文件名
- token 估算
- 摘要和关键词

#### 阶段 3：语义标注

由 [rag/indexing/semantic_annotator.py](D:/ai/project/rag/rag/indexing/semantic_annotator.py) 负责：

- 加载标注检查点
- 跳过已完成标注的 chunk
- 并发调用 `qwen3.6-plus`
- 补充 `summary`、`keywords`、`tags`、`type`、`has_code`、`coherence`

#### 阶段 4：向量化

由 [rag/indexing/embedding_client.py](D:/ai/project/rag/rag/indexing/embedding_client.py) 负责：

- 生成正文向量 `content`
- 生成摘要向量 `summary`
- 支持批量请求
- 支持检查点恢复

#### 阶段 5：多路入库

由 [rag/indexing/storage_indexer.py](D:/ai/project/rag/rag/indexing/storage_indexer.py) 负责：

- 写入 Chroma `content_vec`
- 写入 Chroma `summary_vec`
- 构建 BM25 索引
- 生成 `metadata.json`

最终产物：

- `storage/chroma/`
- `storage/bm25.pkl`
- `storage/metadata.json`
- `storage/checkpoints/`

## 5. 在线查询流程

### 5.1 入口

- [rag/cli/answer_query.py](D:/ai/project/rag/rag/cli/answer_query.py)
- [rag/retrieval/query_service.py](D:/ai/project/rag/rag/retrieval/query_service.py)

示例命令：

```bash
.venv/Scripts/python.exe -m rag.cli.answer_query "我要接入谷歌订阅，现在需要如何处理"
```

### 5.2 执行步骤

#### 步骤 1：加载配置与查询服务

`answer_query.py` 会：

- 从环境变量加载 [rag/config/retrieval.py](D:/ai/project/rag/rag/config/retrieval.py) 中的 `QueryConfig`
- 初始化 embedding 模型
- 初始化 Query Rewrite
- 初始化双阶段重排器
- 初始化答案生成器
- 组装 `QueryService`

#### 步骤 2：原始问题三路召回

[rag/retrieval/retriever.py](D:/ai/project/rag/rag/retrieval/retriever.py) 会执行：

- `content_vec` 向量召回
- `summary_vec` 向量召回
- BM25 召回

#### 步骤 3：Query Rewrite 扩召回

若启用 Query Rewrite：

- 使用 `qwen3.6-plus` 生成子查询
- 每个子查询重新执行三路召回
- 将结果合并进总命中集合

#### 步骤 4：候选聚合与规则粗排

- 以 `node_id` 为主键聚合多路命中
- 通过 RRF、多路命中奖励、改写命中奖励和完整度信号进行粗排

#### 步骤 5：邻居扩展

若开启相邻扩展：

- 围绕粗排前若干中心 chunk
- 按 `doc_id + chunk_index` 扩展前后相邻 chunk
- 为后续答案生成补足上下文

#### 步骤 6：LLM 精排

[rag/retrieval/ranking.py](D:/ai/project/rag/rag/retrieval/ranking.py) 会对头部候选做 LLM 精排，默认模型同样为 `qwen3.6-plus`。

#### 步骤 7：答案生成

[rag/retrieval/answer_generator.py](D:/ai/project/rag/rag/retrieval/answer_generator.py) 会：

- 取 Top chunk 作为上下文
- 为每个 chunk 附带文件名、chunk 序号和 `node_id`
- 调用 `qwen3.6-plus` 生成最终答案
- 要求答案末尾附带参考来源

#### 步骤 8：结果输出

命令行最终输出：

- 原始问题
- 改写结果
- 最终答案
- 引用来源（文件名、chunk 序号、`node_id`、分数）
- TopN 候选详情

## 6. 核心配置对流程的影响

| 配置 | 影响 |
| --- | --- |
| `RAG_CHUNK_SIZE` | 影响切分粒度 |
| `RAG_CHUNK_OVERLAP` | 影响上下文连续性 |
| `RAG_ANNOTATOR_MODEL` | 影响离线语义标注模型 |
| `RAG_QUERY_REWRITE_MODEL` | 影响改写模型 |
| `RAG_RERANK_LLM_MODEL` | 影响精排模型 |
| `RAG_ANSWER_MODEL` | 影响答案生成模型 |
| `RAG_RETRIEVAL_NEIGHBOR_ENABLED` | 是否启用邻居扩展 |
| `RAG_ANSWER_CONTEXT_TOP_K` | 进入答案生成的上下文数量 |

## 7. 常见运行顺序

### 第一次建库

1. 预处理原始 Word/Confluence 文档
2. 检查 `cleaned_markdown` 内容
3. 执行离线建库
4. 打开 `chunks_preview.html` 检查切分效果和 `node_id`
5. 执行完整查询验证答案与引用

### 文档更新后重建

1. 重新执行预处理
2. 重新执行离线建库
3. 利用检查点减少重复标注与向量化
4. 重跑典型问题做回归比对

## 8. 故障排查流程

### 8.1 预处理失败

优先排查：

- 原始输入目录
- `soffice` 可用性
- 原始导出格式是否合法

### 8.2 建库失败

优先排查：

- LLM Key / Embedding Key
- 标注失败数量
- 向量化失败批次
- Chroma 写入失败

### 8.3 查询失败

优先排查：

- `storage/chroma` 是否存在
- `metadata.json` 是否存在
- `bm25.pkl` 是否存在
- 查询侧模型配置是否完整
- Query Rewrite、精排或答案生成接口是否异常
